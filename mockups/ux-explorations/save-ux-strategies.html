Save UX — Strategy Comparison
Four candidate save strategies, each shown applied to the three sidebars the user named: Cabinet Builder, Cut List, and Orders. All strategies assume the editing surface is a right-side sidebar (consistent with Cabinet Builder and Cut List today; future migration target for Quote/Order/Client/Project/Stock). Real CSS tokens (--accent, --surface, .cl-unsaved-pill styles) are reused for fidelity.

Strategy A Universal Save Button (Pessimistic)
Every editing sidebar gets a sticky Discard / Save footer. Nothing autosaves silently; an unsaved pill appears in the header until the user clicks Save. Save fires _toast('Saved'). beforeunload blocks tab close while dirty.
Pros: Predictable; one mental model; fewer DB writes.
Cons: Friction for Settings/Business Info; risk users forget to click Save and assume it persisted.

Strategy B Universal Autosave with Status (Notion / Linear)
Every change autosaves on blur (or 600 ms debounce on text). A live status pill in the header cycles through Saving… → Saved → Failed. No save buttons except in creation flows. Failed saves surface a toast and inline retry.
Pros: Zero data loss; modern feel; no friction.
Cons: Heavier DB write load; users may instinctively look for a Save button and feel uncertain.

Strategy C · Recommended Hybrid: Autosave for Editing, Explicit for Creation
Creating a new entity uses an explicit Cancel / + Create footer (so half-typed entities don't pollute lists). Editing an existing entity autosaves with a status pill. Cut List keeps its Save Project snapshot button (it's a versioning action, not a field-edit action).
Pros: Matches user intent (create vs. edit); reuses existing dirty-state infrastructure; minimal friction for routine edits.
Cons: Two patterns to learn (mitigated because they map cleanly to user mental model).

Strategy D Sticky Sidebar Save Bar (GitHub / Linear)
Edits are local-only until the user commits. A sticky bar slides up inside the sidebar when there are unsaved changes, showing the count and Discard / Save buttons. Bar is hidden when clean.
Pros: Visually unmissable when something needs saving; naturally batches writes; one shared pattern.
Cons: Heaviest rework; layout shift when bar appears; awkward for creation flow.

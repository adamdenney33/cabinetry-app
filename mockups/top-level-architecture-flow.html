Top-level architecture flow library-first

Library is the primary workspace · Client tier assembles + bills · Tags categorize · No save-back paths
"Libraries could be all outputs, which simplifies initial workflow. Then added to quotes/clients as needed. Library could use tags for repeat / stock items."
The library tier is the primary workspace. Every cabinet design, cut list, and stock item lives there — and a user can design real work immediately, without needing a client. When a customer arrives, library items get snapshotted into their Quote (the design specs copy into quote_lines; the library item is unchanged). No "save to library" step — every output already is a library item.

Storage stays in three tables (cabinet_templates, stock_items, cutlists-where-client_id-is-null) — they have genuinely different shapes (stock decrements, designs don't; cut lists have FK children; stock has cost/qty fields cabinets don't need). Tags are a UX-layer convention: a tags jsonb column on each table powers cross-cutting filters like "Templates" or "Repeat" without forcing the schema to collapse.

Section 01

Tag categories · the dimensions that replace hard-coded tables

TagTypePurpose · example
cabinet Kind Library item is a cabinet design. Appears in Cabinet tab filter.
cutlist Kind Library item is a cut list pattern. Appears in Cut List tab filter.
stock Kind Library item is physical inventory (has qty, low-alert, cost). Appears in Stock tab filter.
material hardware edge-banding sheet Sub-kind Within stock — what kind of physical thing it is. Drives display category, decimation rules.
template Reusability Marks an item as a reusable template. Default view in Cabinet / Cut List tabs filters for this.
repeat Reusability "I use this often." Pinned / surfaced in the design canvas suggestions.
smith-kitchen Context One-off design tagged with its job. Hides from default template view but findable when needed.
Filter rules: The Cabinet tab's default view is tag = 'cabinet' AND (tag = 'template' OR tag = 'repeat') — shows your reusables. Toggling "All" expands to include one-off designs. The Stock tab is tag = 'stock'. The Cut List tab is tag = 'cutlist' AND tag = 'template' by default.

What's gone: The two reverse-direction "save to library" rows from the previous version. Under library-first, there's nothing to save back — everything is in the library to begin with. Section 03

Why the three tiers still matter

  • Library is upstream of every job. Whether stock material, cabinet template, or cut list pattern — all designs live here without needing a client. Free-tier 5-item cap (per PLAN.md F.2) is enforced on library count.
  • Client tier is downstream assembly. A Quote is "these library items, for this client, at this price, currently in this status." It snapshots specs from the library so client-specific edits don't mutate templates. The source_library_id FK preserves traceability.
  • Derived views own no data. Schedule and Dashboard read the client tier. They never write. Useful for confirming the schedule isn't "a thing we need to migrate" — it follows whatever Orders does.
  • Cabinet tab and Quote tab share a quote row. Per Option E, they're two views of the same quotes row. The Cabinet tab's design canvas is where you either design fresh (writes directly to a quote line) or pull from library (snapshot copies a library item to a quote line).
  • Three tables, one UX. Storage keeps cabinet_templates · stock_items · cutlists(library) separate — they have genuinely different shapes (decrementing stock vs. non-decrementing designs, FK children on cutlists, cost/qty fields on stock). Each gets a tags jsonb column. The "Library tier" is a UX construct, not a single table — tag-driven filters give the user a unified mental model while the schema stays purpose-built.
  • Onboarding gets much simpler. "Design your first cabinet" or "Add your first stock material" — both happen in the library tab the user already opened. No client-creation gate. Free tier users get to play immediately.

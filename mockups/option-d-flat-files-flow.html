Option D — Flat-files model

Client groups everything · Every tab has the same shape · No special "Workspace" concept
"Ultimately, the Client is the thing that groups them."
The proposal: No Project entity. No name-stem matching. No nested "jobs". Quote, Order, Cut List, and Cabinet Project each belong to a Client — every tab has the same shape (list of named files, each containing child rows). Where two specific files need to be connected (cut list generated from a Cabinet Project, quote converted to order), the existing cross-entity FKs already handle that — names don't have to match.

"Where necessary, there is already functionality to link tabs even if names are different — e.g. cutlist to cabinets, or cabinets to quotes, or quotes to orders."
"Treat each cabinet project as a single entity, same as quotes / orders / cut lists."

Section 08

Phased landing path

  1. Phase 1 (1 week) — Option A first. Collapse the Projects tab into the Client card. No schema changes. This proves the simpler IA in production with zero data risk and gives you the empty Projects-tab slot on the nav to confirm nobody misses.
  2. Phase 2 (~3 days) — Add link-chip UI. The .link-bar above editor bodies. Reads from existing FKs (orders.quote_id, cutlists.cabinet_id). Zero schema work — pure UI on top of existing relationships. Once this is in, users can already navigate file→file without a project hub.
  3. Phase 3 (~1 week) — Rename projects → cabinet_projects. Cabinet tab UI now lists "Cabinet Projects" as rows (same shape as Quotes/Orders/Cut Lists). Other tabs lose the project-picker entirely. projects table rename + Cabinet Builder copy pass; no other schema change yet.
  4. Phase 4 (~1 week) — Drop project_id from quotes/orders/cutlists. Backfill client_id on cutlists (other two already have it). Soft-deprecate project_id column for one release. By the end of this phase the model is fully flat.

Total: ~3 weeks across 4 phases, each phase independently shippable and reversible. Phase 1 is the only one that's strictly necessary for the UX win; phases 2–4 progressively clean up the model under it.

Section 09

Risks & mitigations

  • "How do I see all of Jane Smith's kitchen work in one place?" Use the Cabinet Project as the de facto root of a job — its link-chips list every Quote and Cut List exported from it. Or rely on the Client card's drill-in: "Quotes (4)" → filtered Quotes list shows all her quotes sorted by date.
  • Files without a source Cabinet Project lose their connection. A standalone Quote (not created from a Cabinet Project) has no link-chip back to anything except the Client. Fine — that's correct behaviour. It's still findable through the client.
  • Schema migration drops project_id. Bounded risk. Migration runs in phase 4 only after the new model is proven; project_id stays as a soft-deprecated column for one release.
  • [CB_DRAFT] notes-tag pattern goes away. Today cabinets live as quote_lines with a notes-tag flag. After: cabinets live in cabinet_projects directly. One-shot migration translates existing draft quotes → Cabinet Project rows.
  • Dashboard widgets reference "projects". Replace with Cabinet Projects (per-client) or aggregate file counts. Same SQL shape, different table.
  • Naming the entity. The parent record needs a label. Cabinet Project matches your verbal framing, doesn't reintroduce a global Project entity (the word is constrained to the Cabinet context), and parallels "Cut List" as a compound-noun entity name. If you'd prefer Cabinet Set / Cabinet Build / just Cabinet, a single rename pass swaps it.

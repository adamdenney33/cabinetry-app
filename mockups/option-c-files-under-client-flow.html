Option C — Files-under-client model

Quote / Order / Cut List are the job · Cabinet Builder is the only thing with a Workspace · No standalone Project entity in the UI
The proposal: Stop treating "Project" as a parent of all work. Instead, each file (Quote, Order, Cut List) carries its own name and links straight to the client. The "Smith Kitchen" identity emerges from the file name, not from a separate Project row. Only Cabinet Builder keeps a project-shaped container — renamed Workspace — because it produces N cabinets in flight before any are committed to a quote or cut list.

Section 04 · Flow B

Cabinet Workspace → exports to Quote & Cut List

This is the flow where Option C earns its keep. Workspace is the only entity that genuinely needs a "project container" — it produces N cabinets that may fan out to a quote and a cut list. The exports already exist in code; the user model now matches.

Current conceptual ambiguity
  1. 1
    Open Cabinet
    Pick project "Smith Kitchen". The "project" is the cabinet workspace.
  2. 2
    Add 6 cabinets stored as [CB_DRAFT] quote_lines
    Under the hood, cabinets are written to a draft quote tagged [CB_DRAFT]. Same project_id as the eventual real quote.
  3. 3
    "Create Quote from cabinets"
    Copies the draft lines into a new customer-facing quote. Both quote and draft live under the same project.
  4. 4
    Export to Cut List parts copied
    _clLoadCabinetParts + cutlists.cabinet_id FK. But what cabinet? Cabinets are draft quote_lines, not a first-class entity.
  5. 5
    All linked via project_id hub-and-spoke
    Conceptually muddy: cabinets are line-items, workspace is a project, but cabinet-library entries (templates) live in their own table.
Option C model matches reality
  1. 1
    Open Cabinet
    Pick or create Workspace "Smith Kitchen". Workspace is a first-class entity with its own name + client_id.
  2. 2
    Add 6 cabinets in workspace
    Cabinets live in the workspace, not in a draft quote. No [CB_DRAFT] tag fiction.
  3. 3
    Click "→ Quote" direct export
    Creates a Quote with name = workspace.name, client_id = workspace.client_id, lines copied from cabinets. Quote stands on its own — no project layer.
  4. 4
    Click "→ Cut List" direct export
    Creates a Cut List with same name + client; cutlists.workspace_id FK preserves the link back to source workspace.
  5. 5
    Quote → Order via convertQuoteToOrder
    Order inherits the name from the quote. Schedule picks up the order.

Section 06

A pragmatic landing path · do Option A first, then Option C

The two proposals aren't mutually exclusive. Option A is the cheap UX win you can land in a week; Option C is the structural cleanup you do once Option A proves the simpler model works for real users.

Today

9 tabs, Projects hub with 2-state gate, cabinets live in draft quotes via [CB_DRAFT] tag.

  • Project entity is its own row
  • Dedicated Projects tab
  • Cabinets stored as quote_lines
  • 5 tables FK to projects

Step 1 · Option A

Collapse the tab. No schema changes. Projects entity stays, just expanded inline in Client cards.

  • 8 tabs (Projects tab removed)
  • Projects render inline in Client view
  • Existing FK plumbing unchanged
  • ~300 LoC change in clients.js
  • Ship in 1 week.

Step 2 · Option C

Once Option A is proven: drop the project entity from quotes/orders/cutlists; promote Cabinet Builder to a first-class Workspace.

  • Schema migration: add name, drop project_id
  • Rename projects → cabinet_workspaces
  • Drop [CB_DRAFT] pattern; cabinets live in workspace
  • "Job" becomes a visual grouping by name in Client view
  • Plan ~2–3 weeks.
Section 07

Risks & mitigations

  • Name typos break visual grouping. "Smith Kitchen" vs "Smith Kitcen" → no grouping. Mitigation: never make the user retype it. Conversion/export always inherits the name. "+ New Quote" from a client card auto-fills the name from a smart-input over the client's existing job names.
  • Quote revisions get awkward. "Smith Kitchen Quote v1 → v2". Mitigation: either let names handle it ("Smith Kitchen v2") or add a parent_quote_id FK for explicit revision chains. Probably the former — keeps it simple.
  • Cabinet Workspace renaming touches a lot of UI. "Project" appears in Cabinet Builder labels, settings, save-buttons. Mitigation: single search-and-replace pass through Cabinet Builder; user-facing copy only — no logic changes inside cabinet.js.
  • "All work for Jane Smith" reporting. Today: sum project values. After: sum across her quotes/orders/cutlists/workspaces. Same query, just different join.
  • Dashboard "Active Projects" widget. Currently doesn't exist; would need to become "Active Jobs" — derived from open quotes + orders grouped by name. Slightly more work than a simple SELECT * FROM projects but produces the same answer.
  • Schema migration risk. Real but bounded. project_id can stay as a soft-deprecated column for one release while the new name + client_id fields prove out. Roll-forward in a single migration; no data loss because the project name is preserved on every related row.
  • The [CB_DRAFT] pattern unwinds. Today: cabinets live as quote_lines with a [CB_DRAFT] notes tag. After: cabinets live in cabinet_workspaces with their own rows. Mitigation: migrate existing [CB_DRAFT] quotes to workspaces; the data shape is similar enough that this is a routine transform.

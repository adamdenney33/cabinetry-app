Adding library items to a Quote · UI patterns

Five entry points · smart-input picker as the primary mechanism · attribution chip after add
The question: Given the library-first model (cabinet designs, cut list templates, stock items all live in the library tier), what's the actual UI for pulling a library item into a Client's Quote? This doc shows five candidate patterns, recommends one as primary, and walks through the after-add experience.

Anchoring principle: the app already uses a smart-input picker for clients-on-projects, projects-on-quotes, and stock-on-quote-lines (per CLAUDE.md "Library / Data Inputs Pattern"). The library-first model should reuse and extend that pattern — not invent a new one.
Section 01

Five entry points · where the user triggers "add from library"

All five share the same destination (a library picker), but the trigger varies by context. Reuse a single picker component; vary the affordance.

#Entry pointLives inBest forTrigger
1 Inline smart-input Quote · Lines view · "Add line" row Daily building of quotes — type and add fast Smart-input wrap with placeholder "Search library or type to add custom…"
2 "+ from library" tile Quote · Cabinet view · design canvas Designer flow — visual canvas with cabinet thumbnails Dashed-border tile in the canvas grid, same shape as cabinet tiles
3 Toolbar "Browse library" button Quote editor toolbar Browsing mode — don't know what you want yet Opens larger picker modal with grid + sidebar filters
4 Empty-state CTA New / empty Quote First-time orientation — what do I do here? Big buttons: "Add from library" + "Design new"
5 Library tab → "Send to quote" Cabinet / Cut List / Stock tab (library mode) Power-user batch flow — select 6 items, push to a quote Multi-select checkboxes + bottom action bar

Section 05 · Pattern 5

Library tab → "Send to quote" (same shape as Optimize Cut List)

The user is browsing in the Cabinet / Cut List / Stock tab (library mode), spots items they want for a client, multi-selects, and pushes them to a quote. The trigger is from the library side; the destination is the chosen quote.

↺
Familiar motor pattern. This is the same shape as Optimize Cut List: prepare a selection → fire a verb in the bottom action bar → see results in another view. Users who already use Optimize will grok this immediately — no new mental model required.

Comparison & recommendation

PatternDiscoverabilitySpeed (existing users)Multi-addVisual / fastExisting in app?

2 · Canvas tile High (in design grid) Two clicks Multi-select grid Visual — thumbnails, type filters Adapt — cabinet builder has search now

4 · Empty-state CTA High for new quotes only Two clicks via picker — No — needs design
5 · Library tab push ★ familiar shape High — mirrors Optimize Cut List motor pattern Slower for one-off; fastest for batch Batch native List + filters Adapt — Optimize establishes the convention
★
Two primaries, not one. Pattern 1 (inline smart-input) is primary for task-driven work — you're inside a quote and need another line, keyboard-fast. Pattern 5 (library batch send) is primary for inventory-driven work — you're in your library and want to assign items to a quote, same motor pattern as Optimize Cut List. Pattern 2 (canvas tile) is the primary inside Cabinet view's design canvas. Patterns 3 + 4 are optional secondary affordances that can ship later.

Shipping order

  1. Phase 1 — Inline smart-input (Pattern 1). Extends _oStockSearch / _qStockSearch to a unified _smartLibraryForQuoteSuggest that searches all three library tables. Adds attribution chip on resulting lines. ~3 days. Ships independently.
  2. Phase 2 — Library tab batch send (Pattern 5). ★ Familiar shape Multi-select checkboxes in library list views + Optimize-style bottom action bar + destination-quote dropdown. ~3 days. Promoted in shipping order because it mirrors the Optimize Cut List motor pattern users already know — high discoverability without a new mental model.
  3. Phase 3 — Canvas tile (Pattern 2). "+ from library" tile in Cabinet view's design canvas. Opens a thumbnail picker. ~2 days.
  4. Phase 4 — Toolbar + empty state (Patterns 3 + 4). Both call the same browse-modal component from Phase 3. ~2 days. Lowest priority — Patterns 1 + 5 already cover both task-driven and inventory-driven flows.
Section 07

Attribution + override behavior

Every quote_line carries provenance — where it came from. Editing the line in the quote doesn't sync back to the library; the chip stays as a "snapshot of …" marker.

StateChip shownWhat it meansClick chip → ?
Snapshot · unchanged ↰Base 600 template Line is a snapshot from library; specs match the source exactly Side-drawer shows the source library item (read-only); option to "re-snapshot" if library has changed
Snapshot · overridden ↰Base 600 · modified Line started as a snapshot but the user has edited at least one field Drawer shows diff between line and source; option to "revert to template" or "promote changes to library"
Custom · not in library ✎Custom · not in library Line entered fresh (e.g. via "+ Design new" or smart-input "+ Create custom") Opens a "Save to library" prompt: "Promote this to a reusable template?"
Promoted ↛Promoted to library User clicked "Save to library" on a custom line; library now has this design as a template Drawer opens the new library item · option to add a "repeat" tag · view other quotes that use this template

Schema: Each quote_lines row gains two new columns: source_kind text (one of 'cabinet_template' · 'stock_item' · 'cutlist_template' · NULL) and source_id bigint (FK to whichever library table). NULL on both = "Custom · not in library."

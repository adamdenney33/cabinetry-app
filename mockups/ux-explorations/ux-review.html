ProCabinet.App — UX Consistency Review

11 May 2026 · Tactical scope only · No new IA, no schema changes

The app has nine tabs and the pre-launch refactor has already pulled the big interaction patterns into shape — every entity tab opens its records in the sidebar editor, not a modal. The drift that remains is smaller and more specific: card layout, save mechanic, unsaved-state visibility, and foreign-key clickability.

One naming gotcha to flag up front — functions like _openClientPopup / _openStockPopup / _openOrderPopup are legacy aliases. They sound like they open modals; they actually call switchSection() + edit{Entity}() or load{Entity}IntoSidebar(). The drill-in is uniformly the sidebar.

Mockups in the left column are verbatim reproductions of the live app's classes; the right column shows the suggested unified version using the same design tokens.

Section 01

Tab access & drill-in behavior

Drill-in is already unified across all entity tabs — clicking a card or row opens the record in the sidebar editor of that tab. The remaining inconsistencies are smaller: sidebar gate state, sub-tab convention, schedule's twin click models, and the misleading _open*Popup function names.

  • Drill-in is consistent. Every entity tab loads the clicked record into the sidebar editor of the same tab. The _open*Popup function names are misleading — they're all aliases that call switchSection() + edit{Entity}().
  • Schedule has two click models. Calendar bar: single-click opens the order. Sidebar list: single-click scrolls the calendar, double-click opens the order. (schedule.js:174 · schedule.js:289.) Every other tab in the app opens an editor on single-click.
  • Projects has a 2-state sidebar gate. When no client is picked the sidebar shows a "pick client" picker + recent-clients list (_renderProjectsSidebarGate · clients.js:377). Clients/Stock/Quotes/Orders don't have a gate — sidebar is always ready to take input. Justifiable for Projects (every project needs a client) but worth flagging.
  • Sub-tab convention is split three ways. Cut List has a clean 3-tab strip at the top of the main panel. Cabinet has 2 sub-tabs in the sidebar (Builder/Rates) and 2 in the main panel (Project/Library). Quotes/Orders treat the line-kind buttons (Cabinets/Items/Labour) as informal pseudo-tabs.
  • Filter UI varies per tab. Status tabs (Quotes/Orders/Schedule), category pills (Stock), filter+sort dropdowns (Projects), sort-only (Clients), pure search (Cabinet/Cut List), nothing (Dashboard).
  • Function names mislead readers. _openClientPopup / _openProjectPopup / _openStockPopup / _openOrderPopup / _openQuotePopup all look like they open modal popups. They don't — they switch tab + load into sidebar. Rename to _load{Entity}Editor would match what they do.
Proposal · normalise Schedule click model

Make the Schedule sidebar list match every other list in the app: single-click opens the order editor. The "scroll to bar" affordance becomes a separate icon button on the row, or fires automatically when you open the editor.

  • Class names lack convention. .proj-card / .quote-card / .order-card / .stock-row / .card / .dash-row — every entity invented its own, plus two cards have no class at all (Client card, Cut-List library card use inline-styled divs).
  • The .proj-strip action-strip pattern is the de-facto winner but applied inconsistently. Project has 4 cells (related entities), Quote has 3 (line-kinds — different concept!), Client has 1 (Projects only), Order has none, Cut-List has none, Stock has none.

Sidebar interface for creating & saving

Save mechanics split into two camps. Half the tabs autosave silently; the other half need an explicit "+ Add" click. Only Projects shows a visible unsaved-state indicator.

  • Two different save models in one app. Clients and Stock require an explicit button click (saveClientEdit at clients.js:176, saveStockEdit at stock.js:600). Every other tab autosaves. Users learn the wrong mental model first if they land on Clients before Quotes.
  • Only Projects shows save state visually. The .cl-unsaved-pill (● Unsaved → ✓ Saved → ! Failed) is already implemented in styles.css:227 with full animation. Quote / Order / Cabinet / Cut-List autosave silently — users can't tell whether their typing has persisted.
  • Smart-input function names are ad-hoc. Same UI pattern under six different names — sometimes target-first (_smartQProjectSuggest), sometimes source-first (_smartClientSuggest), sometimes library-style (_smartCBLibrarySuggest).
  • Post-save behaviour differs. Clients/Stock close the form. Quotes/Orders/Projects keep the editor open — surprising on a brand-new record where the user expects a "saved" confirmation and a return to the list.

Proposal · sidebar shell standard

Every sidebar editor in the app follows the same 3-block layout:

  1. Header — back arrow · entity icon · title · status chip · .cl-unsaved-pill. This is the existing .ed-head from the mockup-J port.
  2. Body — fields grouped in .pf rows. Smart-inputs (.smart-input-wrap + + popup button) for FK relations.
  3. Footer — Duplicate / Delete (when editing); nothing (when creating — autosave handles it).

Gaps · navigation users would expect but doesn't exist
  • Foreign-key names on cards are not clickable. Project, Quote, and Order cards display the client name (and project name on Quote/Order) but as plain text. To get to the client, user must close the editor, switch to Clients tab, search.
  • Editor headers don't drill either. When editing an order, the header shows "Smith Kitchen — Jane Smith" as static text. Same for Quote, Cabinet, Cut List editors.
  • The existing "Linked: CabinetName" text is static. The cut-list library renders it correctly (cutlist.js:4229) but doesn't make it a link. One onclick away from being a real reciprocal nav.
  • Drill-in scope isn't obvious enough. The blue .ph-back header is the only signal that you're in a filtered Quotes/Orders view. New users miss it and assume their search returns empty because there are no quotes — not because they're scoped to one project.
Proposal · clickable foreign-key names

Project and client names become .ec-link spans wherever they appear on cards and editor headers. With event.stopPropagation() on cards, the card click still opens the editor; clicks specifically on the FK name drill into the FK's tab.

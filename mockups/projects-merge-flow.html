Collapse Projects tab into Clients

Option A · Same entity model · One fewer tab · Single-flow user journey
The proposal: Keep the projects table and every FK (6 child tables, 9 source files). Drop only the dedicated tab + its 2-state sidebar gate. Projects become an expandable sub-list inside each Client card. Drill-in from a project (Cabinets / Cut Lists / Quotes / Orders) keeps working unchanged — the action strip just moves down one level.

Section 07

Risks & counter-arguments

  • "I want a list of all active projects across clients." Solved by a Dashboard widget. If that's not enough, add a "Projects" sub-tab on the Dashboard, not a top-level tab.
  • "Power users with 50+ projects." Inline expansion gets noisy. Mitigation: collapsed-by-default on cards with >5 projects, with a "Show all" expander. Cards with 0–5 projects stay always-open.
  • "Project-only operations (rename, archive, delete)." Stay in the inline project form / row context menu — no functionality lost, just relocated.
  • "What if a project becomes orphaned (client deleted)?" Today: on delete cascade drops it. Same after the change. Cascade rules don't move.
  • Code surface change is larger than the visual change suggests. renderProjectsMain is ~150 lines that fold into renderClientsMain; _pickClientForProjects, _exitClient_projects, the 2-state gate logic, _renderProjectsSidebarGate all collapse. Total: ~300 lines removed, ~200 added inside clients.js. Not trivial — but no schema work, no migration, no data risk.

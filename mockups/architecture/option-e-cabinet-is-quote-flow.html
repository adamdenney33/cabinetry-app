Option E — Cabinet IS Quote

One entity · Two views (Design canvas + Line items) · No [CB_DRAFT] workaround · One fewer table
"What if Cabinet Projects were just Quotes (shared with quotes)?"
The realization: Cabinet designs are already stored as quote_lines with line_kind = 'cabinet'. The current "Cabinet Project" / "Workspace" concept is just a draft quote tagged [CB_DRAFT] in its notes field — a workaround for the fact that we treated cabinet-design as a separate entity in the UI when the data model already collapsed them.

Remove the workaround. A Quote is a Quote. The Cabinet tab is a design-canvas view of a Quote; the Quote tab is the line-items + status view. Same row, same table, two ways to look at it. [CB_DRAFT] disappears. The Cabinet Builder snapshot-to-quote step disappears. One fewer entity in the model.

"Ultimately, the Client is the thing that groups them."
"Treat each cabinet project as a single entity, same as quotes / orders / cut lists."
Section 01

The realization · cabinets already live in quote_lines

Three sequential observations from PLAN.md and the codebase that, taken together, mean we don't need a "Cabinet Project" entity at all.

Observation 1. Per PLAN.md § D.3: "Real cabinet instances inside a project are written to quote_lines."

Today's data model already treats cabinets as line items, not as a separate "cabinet" entity.
→
So cabinets aren't a distinct entity — they're just quote_lines with line_kind = 'cabinet'.
Observation 2. Cabinet Builder edits a "draft quote" tagged [CB_DRAFT] in its notes field. "Create Quote from cabinets" snapshots the draft into a fresh quote.

So there are two quote rows for every job: one tagged draft (the workspace) and one customer-facing (the snapshot).
→
This is a workaround for treating Cabinet Builder as a separate entity when it isn't. Drop the workaround: the design IS the quote. No snapshot. No duplicate row.
Observation 3. A Quote in the Quote tab and a "Cabinet Project" in the Cabinet tab have the same shape: parent record with a name, client_id, status, and N line items.
→
Same shape, same data, same table. Cabinet tab and Quote tab are two views of one entity — not two entities.
Conclusion: Phase 4.2 (2026-05-05) already removed cbSavedQuotes as a separate source of truth. This proposal completes that direction by removing the in-memory / [CB_DRAFT] separation entirely. cbCreateQuoteFromDraft stops being a snapshot — it becomes a status flip.

Section 07

Phased landing path

  1. Phase 1 (1 week) · Option A first. Collapse Projects tab into Client cards. No schema changes. Proves the simpler IA with zero data risk.
  2. Phase 2 (~3 days) · Add link-chip UI. The .link-bar above editor bodies reads existing FKs (orders.quote_id, cutlists.cabinet_id). Pure UI; no schema work.
  3. Phase 3 (~1 week) · Add 'designing' status to quotes. Migrate existing [CB_DRAFT]-tagged quotes to status = 'designing'. Strip the notes-tag. Cabinet tab now lists quotes filtered to status = 'designing' (or all quotes — design decision).
  4. Phase 4 (~3 days) · Wire view-toggle inside editor. "Design" / "Lines" buttons in both Cabinet and Quote tab editors. Same data, different render. Cabinet-tab "Create Quote" button removed (already a quote).
  5. Phase 5 (~1 week) · Rename cutlists.cabinet_id → cutlists.quote_id. Backfill from existing data. Update _clLoadCabinetParts and related call sites.
  6. Phase 6 (~3 days) · Drop legacy. Drop unused cabinets table (was reserved for cabinet projects in Option D; not needed here). Drop project_id from quotes / orders / cutlists after soft-deprecation period.

Total: ~3.5 weeks. Phases 1–2 are common with Option D's landing path; phases 3–6 diverge to fold Cabinet Builder into Quote. Each phase is independently shippable. Order tab and Cut List tab remain untouched throughout.

Section 08

Risks & mitigations

  • Two tabs editing the same row. User opens Smith Kitchen in Cabinet tab on one device, edits dimensions; opens Quote tab on another device, edits markup %. Mitigation: existing autosave + last-write-wins is fine — they're disjoint fields in quote_lines vs quotes. Server-side, the writes don't conflict. UI: a subtle "edited in other view" toast if a stale render is detected.
  • "Should the Cabinet tab show quotes without cabinet lines?" Open design question. Option E-1: Cabinet tab lists all quotes; opening one in design view shows an empty canvas + "Add Cabinet" prompt. Option E-2: Cabinet tab filters to quotes with line_kind = 'cabinet' count > 0. Recommend E-1 (simpler; user can start designing from any quote).
  • Quote without cabinets (pure items + labour). Lives in Quote tab naturally; appears in Cabinet tab with empty canvas. Fine — user adds cabinets if they want, or stays in Lines view.
  • The "status pipeline" gets longer. Adding designing as a pre-draft state extends the existing 3-step pipe (Draft / Sent / Approved) to 4. The .oc-pipeline renderer already iterates through QUOTE_STATUSES — extend the array, no UI rewrite.
  • Existing draft quotes in production. Migration: UPDATE quotes SET status = 'designing' WHERE notes LIKE '%[CB_DRAFT]%'; UPDATE quotes SET notes = TRIM(REPLACE(notes, '[CB_DRAFT]', '')); Idempotent and reversible.
  • Cabinet Builder UI doesn't change much. The canvas, the dimension inputs, the materials picker — all stay. What changes is the wrapper: instead of "save to draft → snapshot to quote", it's "you're already editing the quote; flip status when ready".
  • One genuine simplification, not three options layered. The big mental shift is "Cabinet tab is a view, not an entity owner". Once that lands, the schema cleanup follows naturally.

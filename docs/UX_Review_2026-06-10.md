## P0 — broken or trust-damaging

### 1. "Create Order" can half-fail silently and double-create
- **Observed:** one click on *Create Order* marked the quote `accepted` (PATCH 200) but **never created the order** — no `POST orders`, no error toast, no navigation, nothing in Sentry. A later conversion succeeded, and the combination left **two orders (ORD-0001, ORD-0002) for one quote**, both $138, both on the schedule and dashboard (shots 47–49).
- **Why it matters:** the natural user reaction to silence is to click again → duplicate orders, double-counted pipeline value. And the half-state (accepted quote, no order) strands the user: the card's button only changes to "✓ View Order" once a linked order exists.
- **Likely cause:** `convertQuoteToOrder` flips the quote status **before** creating the order, and an exception between the two (probably sync `quoteTotal(q)` on a quote whose `_lines` cache isn't hydrated) is swallowed as an unhandled promise rejection.
- **Fix:** (a) reorder — create the order first, flip status after; (b) wrap the whole function in try/catch with an error toast + `_setSaveStatus('order','failed')`; (c) idempotency guard — if `_oq` already links an order, confirm before creating another; (d) add `window.addEventListener('unhandledrejection', …)` → Sentry capture so this class of bug is never invisible again.

### 2. New users see unexplained demo data for ~5–6 seconds
- **Observed:** after first sign-in the dashboard paints **fully populated** (Active Orders ORD-0315…, stock alerts, revenue chart) ~5–6 s before the tour intro modal appears to explain "we've loaded a sample project" (shots 09 vs 13).
- **Why it matters:** the very first impression is "whose data is this?" — on a slow connection the gap is longer, and a user who reflexively dismisses overlays may never read the explanation at all.
- **Fix:** any of — show the tour intro immediately on the auth event (before demo data paints); skeleton-load until the modal is up; or watermark demo entities (e.g. a "SAMPLE" ribbon on cards/rows) whenever `_demoMode` is on outside the tour.

### 3. Account deletion is impossible via cascade (ops/GDPR)
- **Observed during cleanup:** `delete from auth.users …` fails — `orders_user_id_fkey` (and others) have **no ON DELETE CASCADE**. Some tables cascade, some don't; a user-deletion request today requires hand-written table-by-table SQL.
- **Fix:** migration normalising every `user_id` FK to `on delete cascade` (SCHEMA.md § patterns), or a `delete_user_data(uid)` SECURITY DEFINER function. Needed before public launch — UK/EU users have deletion rights.

## P1 — flow friction in the core loop

### 4. Quote → "+ Cabinet" lands behind a second gate
Clicking **+ Cabinet** in the quote editor switches to the Cabinet tab… where you must click **+ Add Cabinet** again (shot 37 → 38). Intent was already declared; open a fresh cabinet directly (the gate's insta-create-with-defaults pattern is already there).

### 5. Cut List doesn't pull from the quote you just built
The Cut List tab's empty state offers only "+ Add Cut List" / library jargon ("Use *Add to Library* under Optimize…"), with no path from existing quote/order cabinets (shot 51). The only bridge is the small per-cabinet "Link to Cutlist 0 +" control in the builder. Add a "Start from a quote's cabinets" action to the cut-list empty state, and rename the builder control (e.g. "Send to cut list").

### 6. Live link: stuck/ambiguous creation state + contradictory copy
"Creating live link…" appeared in two places and hadn't resolved seconds later; the preview pane simultaneously says "Once this quote **has items** and its live link is ready…" on a quote that has an item (shot 44). Also: merely opening the Live link pane flipped the quote to **Sent** — status change from a *view* action will surprise people (consider flipping on first customer visit instead, or at least toasting "Quote marked as sent").

### 7. Two different default-tax paths
"+ Add Quote" creates with `tax: 0`; the editor-create path defaults `tax: 13` (DB default — looks like Canadian HST). UK/AU users (your actual early users) get either 0% or 13% silently — both wrong. Unify, and source the default from Business Info with a locale-aware suggestion (UK 20 / AU 10) during onboarding.

## P2 — UI polish

8. **"SAVED" badge on unsaved forms** — the New Client (and similar) sidebar shows a green SAVED pill before anything is saved (shot 27). Hide it until the first persist, or show "Draft".
9. **Zero badges render** — nav shows "Orders 0" after the demo ends (shot tour-end); badge should disappear at 0 (demo's "Stock 3 / Orders 5" behave correctly).
10. **Recent Quotes empty card is blank** — siblings have copy ("No active orders", "All materials well stocked"); Recent Quotes shows whitespace (shot tour-end).
11. **Empty revenue chart looks broken** — y-axis ticks "1, 1, 0" with ghost bars when there are no completed orders (shots tour-end, 49). Swap for an empty-state line.
12. **Rates panel a11y** — no programmatic `<label for>` association anywhere (labels read as adjacent text nodes; inputs have no ids). Screen-reader users get unlabeled number fields (shot 20).
13. **Schedule sidebar touch targets** — the ▲▼ priority arrows are ~10 px and order rows truncate hard ("ORD-0002 · ZZ Test …" + due "—") (shot 48). Min 24 px targets; show due date or drop the dash.
14. **Calendar opens two weeks in the past** — initial scroll starts at late May for a June 10 today (shot 48); start the viewport at the current week (Today button exists but shouldn't be needed on first paint).
15. **Mobile tab bar is icon-only** — eight unlabeled glyphs (shot 53/54). Standard practice is icon + tiny label on bottom navs; at minimum label the active tab.
16. **Quote/order card action rows are busy on mobile** — six buttons wrap to two lines (shot 54); consider collapsing secondary actions (Duplicate/Delete/Messages) into the ⋯ overflow on small screens.

## P3 — nits

17. Landing: the sticky strip (Dashboard / Cut List / …) mimics the app's tab bar; as scrollspy anchors it works, but first-time visitors may read it as a broken app nav. A "Features:" prefix or pill styling would disambiguate.
18. Landing fine print: 21 text nodes under 11 px (legal/sub-notes) — fine, but check contrast on the grey-on-white ones.
19. Annual pricing copy "$15/mo · $180 billed year one, then $299/yr" — the year-2 jump reads as a price rise mid-sentence; consider "then $299/yr (still $25/mo equiv.)".
20. Stock add form is 11 fields up front; essentials (name, size, cost) + an expandable "more" would speed the common case.

## What's notably good (keep)

Instant-create with sensible defaults everywhere (cabinet appears priced at $108 the moment you add it); the live price breakdown (materials/labour/contingency/markup) is the product's wow moment and it works; smart-input + suggest pattern is consistent across clients/stock/quotes; guest demo mode is a great try-before-signup; pipeline cards with inline status; Getting Started card's 4-step aha path with progress ticks; tour is tight at 10 steps with honest copy; autosave status pills; no horizontal overflow on mobile anywhere tested.

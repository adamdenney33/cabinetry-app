Project Context — Strategy Comparison
Today the Project field competes for visual hierarchy with Cabinet, Stock, and Edge-Banding libraries on the same sidebar — it looks like one input among many. The four strategies below explore how "being inside a project" feels, biased toward the Quotes/Orders "in-a-folder" pattern you flagged as the strongest model. Each strategy is shown across the four affected tabs: Cabinet Builder, Cut List, Quotes, and Orders.

Baseline · Today

Cabinet Builder and Cut List sidebars stack a Project smart-input alongside Cabinet/Stock/Edge libraries with similar visual weight. Quotes and Orders already break this pattern — they swap the picker for an "Editing: Project Name" chip + dirty pill once a project is loaded, and the rest of the sidebar becomes a project-scoped editor. The strategies below borrow from that and apply it back to Cabinet Builder and Cut List, with four levels of commitment.
Strategy 1 Project Chip Header (lightest touch)
When a project resolves, the Project smart-input collapses into a chip with the project name + meta + save status. Clicking the chip opens a dropdown to switch / exit. Other libraries stay inline below the chip with a clear visual demotion (smaller labels, scoped-section dividers).
Pros: Smallest delta from current layout; preserves all existing inline tools; chip is visually distinct from input fields so the "muddy waters" problem is partly solved.
Cons: Libraries still compete for attention below the chip; doesn't fully feel like you're "inside" the project.

Strategy 2 Gated Sidebar — empty state vs in-project state
The sidebar has two modes. Empty: a centered project picker + recent-project list with no other libraries showing. In-project: a project header with a ← Back link and project-scoped tools below. The libraries don't appear until you've committed to a project, so they can't compete for attention.
Pros: Strongest "inside vs outside" feel; libraries are clearly scoped to the active project; matches mental model of "open a project, work in it, close it."
Cons: Empty state takes a click to escape (back to picker); slightly heavier modal-feel; existing Cabinet Builder users used to typing the project inline lose that.

Strategy 3 · Recommended Workspace-level Project — project lives in the app header
Lifts the project picker out of every sidebar and into the app header, alongside the logo. Like Linear's workspace switcher: pick a project once and every tab (Cabinet Builder, Cut List, Quotes, Orders) operates within that project's context. Each tab's sidebar drops the project field entirely and is left with just its own libraries / tools.
Pros: Cleanly solves the "muddy waters" problem — project is hierarchically above libraries, not next to them; tab-switching keeps the same project (huge usability win); matches mental model of established multi-tab project tools.
Cons: Heaviest reshape — touches app shell + every tab; user-level data (Stock, Settings) needs a clear "no project" or "all-projects" distinction; existing keyboard muscle memory of typing project name on each tab is gone.
Why we recommend it: the user-noted issue isn't just visual fidelity — it's that the project keeps showing up as a peer of libraries on every tab. The cleanest answer is to lift project up one level so it stops being a tab-local concept. This also matches how Quotes and Orders already imply the project is a containing context.

Strategy 4 Project Containment — sidebar = project header + tools
Mirrors the Quotes/Orders pattern wholesale. The sidebar becomes a project header + add-tools panel; libraries (Cabinet, Stock, Edge) become collapsible drawer rows instead of inline smart-inputs. The main content area is the project's actual data (cabinets list, cut layout) — feels most "you are inside the project."
Pros: Strongest "in-a-folder" feel; libraries are visually demoted to single-line drawer entries; main content area becomes the focal point.
Cons: Drawer pattern adds a click to access libraries that were previously inline; bigger jump for Cut List which currently has libraries as primary tools.

Pick Recommendation
Strategy 3 (Workspace-level Project) is the cleanest fit for what you described. The reason: lifting project up one level out of the sidebar means it stops being a peer of libraries and becomes the container they live inside. Cabinet Builder, Cut List, Quotes, and Orders all become "tabs of a project" rather than "tabs that each pick a project independently" — and tab-switching keeps the same project, which today requires re-typing it on every tab.

If Strategy 3's app-shell change is too much, Strategy 1 (Chip) is the safe minimum-delta version: same architecture, just makes the project look like a header chip instead of a peer input. Strategy 4 (Containment) is the most "Quotes/Orders-faithful" but adds a click to reach libraries.
